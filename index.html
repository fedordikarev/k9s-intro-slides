Вводный курс

Kubernetes + kubectl + k9s + Helm + Helmfile

Зачем Kubernetes?

  • Оркестрация контейнеров
  • Автоматический деплой, масштабирование, самовосстановление
  • Почему это важно для нас

Архитектура на пальцах

Pod → Deployment → Service → Ingress

Control plane упоминаем только концептуально

Основные сущности

  • Pod: минимальная единица
  • Deployment: декларативный контроль
  • Service: доступ к Pod’ам

Мини-пример

kubectl apply -f deployment.yaml

Что при этом создается в кластере

Подключение к кластеру

  • kubeconfig
  • контексты: kubectl config use-context

Основные команды kubectl

  • kubectl get (pods, deployments, svc)
  • kubectl describe
  • kubectl logs
  • kubectl exec

Пример цепочки: найти pod → посмотреть логи

Практические советы kubectl

  • Флаги: -o wide, -n namespace, --watch
  • Алиасы: k, kgp, kgns

Что такое k9s

  • TUI-интерфейс к кластеру
  • Быстрее и удобнее, чем kubectl

Возможности k9s

  • Навигация по ресурсам
  • Логи и exec
  • Фильтрация и горячие клавиши

Зачем нужен Helm?

  • Проблема: много YAML
  • Chart + Values как шаблон и параметры
  • Повторное использование

Пример работы с Helm

helm install nginx bitnami/nginx

Изменение параметров через values.yaml

Зачем нужен Helmfile?

  • Управление несколькими чартами
  • Декларативный список релизов

Пример Helmfile.yaml

releases:
  - name: nginx
    chart: bitnami/nginx
    values: [values/nginx.yaml]
  - name: redis
    chart: bitnami/redis

Одна команда: helmfile apply

Итоговый сценарий

  1. Посмотреть состояние (kubectl / k9s)
  2. Найти проблему (logs / describe / exec)
  3. Деплой новой версии (helm / helmfile)
